## 系统概述

### 什么是 Superpowers？

Superpowers 是一套用于增强 AI 编程助手能力的技能系统。每个技能都是一个经过验证的最佳实践指南，帮助 AI 在特定场景下做出更好的决策。

### 核心理念

- **规则优先**：技能定义了明确的规则和流程
- **自动触发**：AI 会根据场景自动识别并使用相应技能
- **质量保证**：每个技能都经过实战验证

### 技能分类

1. **流程技能**：定义开发工作流（brainstorming, writing-plans, executing-plans）
2. **质量保证技能**：确保代码质量（test-driven-development, systematic-debugging）
3. **协作技能**：代码审查和反馈（requesting-code-review, receiving-code-review）
4. **工具技能**：辅助工具和方法（using-git-worktrees, verification-before-completion）

## 技能列表

### 1. 🎯 using-superpowers
**使用场景**：每次对话开始时

**核心功能**：
- 建立如何查找和使用技能的基础
- 要求在任何响应（包括澄清问题）之前调用技能工具

**关键规则**：
- 如果有 1% 的可能某个技能适用，**必须**调用该技能
- 这不是可选的，不能合理化逃避
- 技能检查必须在澄清问题**之前**进行

**使用方法**：
```markdown
当收到用户消息时：
1. 检查是否有任何技能可能适用
2. 如果是，调用该技能
3. 宣布："使用 [技能名] 来 [目的]"
4. 严格遵循技能指导
5. 然后回应
```

### 2. 💡 brainstorming
**使用场景**：任何创造性工作之前 - 创建功能、构建组件、添加功能或修改行为

**核心流程**：

#### 阶段 1：理解想法
- 先检查当前项目状态（文件、文档、最近提交）
- **一次问一个问题**来完善想法
- 优先使用多选题
- 关注：目的、约束、成功标准

#### 阶段 2：探索方案
- 提出 2-3 种不同方法及权衡
- 以对话方式展示选项及推荐
- 先展示推荐选项并解释原因

#### 阶段 3：呈现设计
- 一旦理解要构建什么，呈现设计
- 分成 200-300 字的小节
- 每节后询问是否正确
- 涵盖：架构、组件、数据流、错误处理、测试

#### 完成后
1. **文档化**：将验证的设计写入 `docs/plans/YYYY-MM-DD-<主题>-design.md`
2. **提交**：将设计文档提交到 git
3. **实施准备**：
   - 使用 `using-git-worktrees` 创建隔离工作空间
   - 使用 `writing-plans` 创建详细实施计划

**关键原则**：
- **一次一个问题** - 不要用多个问题压倒用户
- **优先多选** - 比开放式问题更容易回答
- **严格 YAGNI** - 从所有设计中删除不必要的功能
- **探索替代方案** - 在确定之前总是提出 2-3 种方法
- **增量验证** - 分节呈现设计，验证每一节

### 3. 📝 writing-plans
**使用场景**：有规格或多步骤任务需求时，接触代码之前

**核心原则**：编写假设工程师对我们的代码库零上下文的全面实施计划

**计划必须包含**：
- 每个任务要接触哪些文件
- 代码、测试、需要检查的文档
- 如何测试
- 以小块任务形式给出完整计划
- 遵循 DRY、YAGNI、TDD、频繁提交

#### 任务粒度
每个步骤是一个动作（2-5 分钟）：
- "编写失败测试" - 一步
- "运行确保失败" - 一步  
- "实现最小代码使测试通过" - 一步
- "运行测试确保通过" - 一步
- "提交" - 一步

#### 计划文档头部
```markdown
# [功能名称] 实施计划

> **给 Claude：** 必需子技能：使用 superpowers:executing-plans 逐任务实施此计划

**目标：** [一句话描述构建什么]

**架构：** [2-3 句关于方法]

**技术栈：** [关键技术/库]

---
```

#### 任务结构示例
```markdown
### 任务 N：[组件名称]

**文件：**
- 创建：`exact/path/to/file.py`
- 修改：`exact/path/to/existing.py:123-145`
- 测试：`tests/exact/path/to/test.py`

**步骤 1：编写失败测试**
[完整代码]

**步骤 2：运行测试验证失败**
运行：`pytest tests/path/test.py::test_name -v`
预期：失败，显示"function not defined"

**步骤 3：编写最小实现**
[完整代码]

**步骤 4：运行测试验证通过**
运行：`pytest tests/path/test.py::test_name -v`
预期：通过

**步骤 5：提交**
```bash
git add tests/path/test.py src/path/file.py
git commit -m "feat: add specific feature"
```
```

#### 执行移交
计划完成后，提供执行选择：
1. **子代理驱动（当前会话）** - 每任务分派新子代理，任务间审查
2. **并行会话（单独）** - 在新会话中使用 executing-plans，批量执行带检查点

**保存位置**：`docs/plans/YYYY-MM-DD-<功能名>.md`

### 4. ⚙️ executing-plans
**使用场景**：有书面实施计划需在单独会话中执行时

**核心原则**：批量执行带检查点供架构师审查

#### 流程

**步骤 1：加载和审查计划**
1. 读取计划文件
2. 批判性审查 - 识别任何问题或疑虑
3. 如有疑虑：在开始前向人类伙伴提出
4. 如无疑虑：创建 TodoWrite 并继续

**步骤 2：执行批次**
默认：前 3 个任务

对每个任务：
1. 标记为进行中
2. 严格遵循每个步骤
3. 按规定运行验证
4. 标记为完成

**步骤 3：报告**
批次完成时：
- 显示实施了什么
- 显示验证输出
- 说："准备好接收反馈"

**步骤 4：继续**
基于反馈：
- 如需要应用更改
- 执行下一批次
- 重复直到完成

**步骤 5：完成开发**
所有任务完成并验证后：
- 宣布："使用 finishing-a-development-branch 技能完成此工作"
- 使用该技能验证测试、呈现选项、执行选择

#### 何时停止并寻求帮助
**立即停止**当：
- 批次中遇到阻碍（缺少依赖、测试失败、指令不清）
- 计划有关键缺口阻止开始
- 不理解指令
- 验证反复失败

**寻求澄清而非猜测**

### 5. 🧪 test-driven-development (TDD)
**使用场景**：实施任何功能或错误修复时，编写实现代码之前

**核心原则**：先写测试。看它失败。写最小代码通过。

**铁律**：
```
没有失败测试就没有生产代码
```

#### Red-Green-Refactor 循环

**RED - 编写失败测试**
- 编写一个最小测试显示应该发生什么
- 清晰的名称，测试真实行为，一件事

**验证 RED - 看它失败**
- **强制性。永不跳过**
- 运行测试确认失败（不是错误）
- 失败消息符合预期
- 因功能缺失失败（不是拼写错误）

**GREEN - 最小代码**
- 编写最简单的代码通过测试
- 不要添加功能、重构其他代码或超越测试"改进"

**验证 GREEN - 看它通过**
- **强制性**
- 测试通过
- 其他测试仍通过
- 输出干净（无错误、警告）

**REFACTOR - 清理**
- 仅在绿色后：删除重复、改进命名、提取辅助函数
- 保持测试绿色。不添加行为

#### 为什么顺序重要

**"我之后会写测试来验证它工作"**
- 代码后写的测试立即通过。立即通过证明不了什么
- 测试可能测试错误的东西
- 可能测试实现，而非行为
- 可能遗漏你忘记的边缘情况

**"我已经手动测试了所有边缘情况"**
- 手动测试是临时的
- 没有测试内容的记录
- 代码更改时无法重新运行
- 压力下容易忘记情况

**"删除 X 小时的工作是浪费"**
- 沉没成本谬误
- 现在的选择：
  - 删除并用 TDD 重写（X 更多小时，高信心）
  - 保留并在后添加测试（30 分钟，低信心，可能有错误）

#### 示例：错误修复

**错误：** 接受空电子邮件

**RED**
```typescript
test('rejects empty email', async () => {
  const result = await submitForm({ email: '' });
  expect(result.error).toBe('Email required');
});
```

**验证 RED**
```bash
$ npm test
FAIL: expected 'Email required', got undefined
```

**GREEN**
```typescript
function submitForm(data: FormData) {
  if (!data.email?.trim()) {
    return { error: 'Email required' };
  }
  // ...
}
```

**验证 GREEN**
```bash
$ npm test
PASS
```

#### 验证清单
- [ ] 每个新函数/方法都有测试
- [ ] 实施前看每个测试失败
- [ ] 每个测试因预期原因失败（功能缺失，非拼写错误）
- [ ] 写最小代码通过每个测试
- [ ] 所有测试通过
- [ ] 输出干净（无错误、警告）

### 6. 🔍 systematic-debugging
**使用场景**：遇到任何错误、测试失败或意外行为时，提出修复之前

**核心原则**：在尝试修复之前始终找到根本原因。症状修复是失败。

**铁律**：
```
没有根本原因调查就没有修复
```

#### 四个阶段

**阶段 1：根本原因调查**

1. **仔细阅读错误消息**
   - 不要跳过错误或警告
   - 它们通常包含确切的解决方案
   - 完整阅读堆栈跟踪

2. **一致地重现**
   - 能可靠触发吗？
   - 确切步骤是什么？
   - 每次都发生吗？

3. **检查最近更改**
   - 什么更改可能导致这个？
   - Git diff，最近提交
   - 新依赖项，配置更改

4. **收集证据（多组件系统）**
   - 在每个组件边界记录输入/输出数据
   - 验证环境/配置传播
   - 检查每层状态

5. **跟踪数据流**
   - 错误值从哪里来？
   - 什么用错误值调用了这个？
   - 继续向上跟踪直到找到来源

**阶段 2：模式分析**

1. **找工作示例** - 在同一代码库中找类似工作代码
2. **对照参考比较** - 完整阅读参考实现
3. **识别差异** - 工作和损坏之间有什么不同？
4. **理解依赖** - 这需要什么其他组件？

**阶段 3：假设和测试**

1. **形成单一假设**
   - 明确陈述："我认为 X 是根本原因，因为 Y"
   - 具体，不模糊

2. **最小测试**
   - 做最小可能的更改来测试假设
   - 一次一个变量

3. **继续前验证**
   - 有效？是 → 阶段 4
   - 无效？形成新假设

**阶段 4：实施**

1. **创建失败测试用例**
   - 最简单的可能重现
   - 如可能自动化测试
   - 修复前必须有

2. **实施单一修复**
   - 处理识别的根本原因
   - 一次一个更改

3. **验证修复**
   - 测试现在通过？
   - 没有其他测试损坏？

4. **如果修复不起作用**
   - 停止
   - 如果尝试 < 3 次：返回阶段 1
   - **如果 ≥ 3 次：停止并质疑架构**

5. **如果 3+ 修复失败：质疑架构**
   - 这表明架构问题
   - 停止并与人类伙伴讨论基本面

#### 常见合理化

| 借口 | 现实 |
|------|------|
| "问题简单，不需要流程" | 简单问题也有根本原因。流程对简单错误很快 |
| "紧急，没时间走流程" | 系统调试比猜测和检查更快 |
| "先试试这个，然后调查" | 第一个修复设定模式。从一开始就做对 |
| "再尝试一次修复"（已尝试 2+ 后） | 3+ 失败 = 架构问题。质疑模式，不要再修复 |

### 7. ✅ verification-before-completion
**使用场景**：声称工作完成、修复或通过之前，提交或创建 PR 之前

**核心原则**：声称前的证据，始终。

**铁律**：
```
没有新鲜验证证据就没有完成声明
```

#### 门函数

```
在声称任何状态或表达满意之前：

1. 识别：什么命令证明这个声称？
2. 运行：执行完整命令（新鲜、完整）
3. 阅读：完整输出，检查退出代码，计数失败
4. 验证：输出确认声称吗？
   - 如果否：陈述实际状态及证据
   - 如果是：陈述声称及证据
5. 然后才：做出声称

跳过任何步骤 = 撒谎，不是验证
```

#### 常见失败

| 声称 | 需要 | 不充分 |
|------|------|--------|
| 测试通过 | 测试命令输出：0 失败 | 之前的运行，"应该通过" |
| Linter 干净 | Linter 输出：0 错误 | 部分检查，推断 |
| 构建成功 | 构建命令：退出 0 | Linter 通过，日志看起来不错 |
| 错误修复 | 测试原始症状：通过 | 代码更改，假设修复 |

#### 关键模式

**测试：**
```
✅ [运行测试命令] [看到：34/34 通过] "所有测试通过"
❌ "现在应该通过" / "看起来正确"
```

**回归测试（TDD Red-Green）：**
```
✅ 编写 → 运行（通过）→ 恢复修复 → 运行（必须失败）→ 恢复 → 运行（通过）
❌ "我已经写了回归测试"（没有 red-green 验证）
```

#### 何时应用

**总是在以下之前：**
- 任何成功/完成声称的变体
- 任何满意表达
- 任何关于工作状态的积极陈述
- 提交、PR 创建、任务完成
- 移到下一个任务

### 8. 👀 requesting-code-review
**使用场景**：完成任务、实施主要功能或合并前验证工作符合要求时

**何时请求审查**

**强制性：**
- 子代理驱动开发中每个任务后
- 完成主要功能后
- 合并到主分支前

**可选但有价值：**
- 被困时（新视角）
- 重构前（基线检查）
- 修复复杂错误后

#### 如何请求

**1. 获取 git SHA：**
```bash
BASE_SHA=$(git rev-parse HEAD~1)  # 或 origin/main
HEAD_SHA=$(git rev-parse HEAD)
```

**2. 分派代码审查子代理**

**3. 处理反馈：**
- 立即修复关键问题
- 继续前修复重要问题
- 记录次要问题供以后处理
- 如果审查者错了推回（有理由）

### 9. 📥 receiving-code-review
**使用场景**：收到代码审查反馈时，实施建议前，特别是如果反馈似乎不清楚或技术上有问题

**核心原则**：实施前验证。假设前询问。技术正确性优于社会舒适。

#### 响应模式

```
收到代码审查反馈时：

1. 阅读：完整反馈不反应
2. 理解：用自己的话重述需求（或询问）
3. 验证：对照代码库现实检查
4. 评估：对这个代码库技术上合理吗？
5. 响应：技术确认或有理由的推回
6. 实施：一次一项，测试每个
```

#### 禁止响应

**永不：**
- "你完全正确！"（明确违反）
- "好观点！" / "优秀反馈！"（表演性）
- "让我现在实施"（验证前）

**相反：**
- 重述技术需求
- 问澄清问题
- 如果错了用技术理由推回
- 只是开始工作（行动 > 言语）

#### 处理不清楚的反馈

```
如果任何项目不清楚：
  停止 - 还不要实施任何东西
  询问不清楚项目的澄清

为什么：项目可能相关。部分理解 = 错误实施
```

#### 何时推回

推回当：
- 建议破坏现有功能
- 审查者缺少完整上下文
- 违反 YAGNI（未使用的功能）
- 对此技术栈技术上不正确
- 存在遗留/兼容性原因
- 与人类伙伴的架构决策冲突

**如何推回：**
- 使用技术理由，不是防御性
- 问具体问题
- 引用工作测试/代码
- 如果是架构问题涉及人类伙伴

### 10. 🌳 using-git-worktrees
**使用场景**：开始需要与当前工作空间隔离的功能工作或执行实施计划前

**核心原则**：系统目录选择 + 安全验证 = 可靠隔离

#### 目录选择流程

**1. 检查现有目录**
```bash
ls -d .worktrees 2>/dev/null     # 首选（隐藏）
ls -d worktrees 2>/dev/null      # 替代
```

**2. 检查 CLAUDE.md**
```bash
grep -i "worktree.*director" CLAUDE.md 2>/dev/null
```

**3. 询问用户**
如果没有目录存在且没有 CLAUDE.md 偏好：
```
未找到 worktree 目录。我应该在哪里创建 worktrees？

1. .worktrees/（项目本地，隐藏）
2. ~/.config/superpowers/worktrees/<项目名>/（全局位置）

你更喜欢哪个？
```

#### 安全验证

**对于项目本地目录：**
```bash
# 检查目录是否被忽略
git check-ignore -q .worktrees 2>/dev/null || git check-ignore -q worktrees 2>/dev/null
```

**如果未被忽略：**
1. 添加适当行到 .gitignore
2. 提交更改
3. 继续创建 worktree

#### 创建步骤

1. **检测项目名称**
2. **创建 Worktree**
3. **运行项目设置**（自动检测：npm install、cargo build 等）
4. **验证干净基线**（运行测试）
5. **报告位置**

---

### 11. 🔄 dispatching-parallel-agents
**使用场景**：面对 2+ 可以无共享状态或顺序依赖工作的独立任务时

**何时使用**

使用当：
- 3+ 测试文件因不同根本原因失败
- 多个子系统独立损坏
- 每个问题可以在没有其他上下文的情况下理解
- 调查之间没有共享状态

不使用当：
- 失败相关（修复一个可能修复其他）
- 需要理解完整系统状态
- 代理会相互干扰

#### 模式

**1. 识别独立域**
按损坏内容分组失败

**2. 创建聚焦代理任务**
每个代理获得：
- 特定范围
- 清晰目标
- 约束
- 预期输出

**3. 并行分派**
```typescript
Task("修复 agent-tool-abort.test.ts 失败")
Task("修复 batch-completion-behavior.test.ts 失败")
Task("修复 tool-approval-race-conditions.test.ts 失败")
// 三个并发运行
```

**4. 审查和集成**
- 阅读每个摘要
- 验证修复不冲突
- 运行完整测试套件

### 12. 🚀 subagent-driven-development
**使用场景**：在当前会话中执行有独立任务的实施计划时

**核心原则**：每任务新子代理 + 两阶段审查（先规格后质量）= 高质量，快速迭代

#### 流程

**每个任务：**

1. **分派实施者子代理**
2. **实施者询问问题？** → 回答，提供上下文
3. **实施者实施、测试、提交、自审**
4. **分派规格审查者子代理**
5. **规格确认代码匹配规格？** → 如否，实施者修复
6. **分派代码质量审查者子代理**
7. **代码质量审查者批准？** → 如否，实施者修复
8. **在 TodoWrite 中标记任务完成**

**所有任务后：**
- 分派最终代码审查者审查整个实施
- 使用 `finishing-a-development-branch`

#### 优势

- 子代理自然遵循 TDD
- 每任务新上下文（无混淆）
- 并行安全（子代理不干扰）
- 两阶段审查：规格合规性，然后代码质量

#### 红旗

**永不：**
- 跳过审查（规格合规性或代码质量）
- 继续未修复问题
- 并行分派多个实施子代理
- 让子代理读计划文件
- **在规格合规性 ✅ 前开始代码质量审查**

---

### 13. 🏁 finishing-a-development-branch
**使用场景**：实施完成，所有测试通过，需要决定如何集成工作时

**核心原则**：验证测试 → 呈现选项 → 执行选择 → 清理

#### 流程

**步骤 1：验证测试**
```bash
npm test / cargo test / pytest / go test ./...
```

如果测试失败：停止。在继续前必须修复。

**步骤 2：确定基础分支**

**步骤 3：呈现选项**
```
实施完成。你想做什么？

1. 本地合并回 <基础分支>
2. 推送并创建 Pull Request
3. 保持分支原样（我稍后处理）
4. 丢弃此工作

哪个选项？
```

**步骤 4：执行选择**

**选项 1：本地合并**
- 切换到基础分支
- 拉取最新
- 合并功能分支
- 验证合并结果上的测试
- 删除分支
- 清理 worktree

**选项 2：推送并创建 PR**
- 推送分支
- 创建 PR
- 清理 worktree

**选项 3：保持原样**
- 报告保持分支
- **不清理 worktree**

**选项 4：丢弃**
- 先确认
- 删除分支
- 清理 worktree

**步骤 5：清理 Worktree**
仅对选项 1、2、4

### 14. ✍️ writing-skills
**使用场景**：创建新技能、编辑现有技能或部署前验证技能工作时

**核心原则**：写技能就是将 TDD 应用于流程文档

#### TDD 映射

| TDD 概念 | 技能创建 |
|----------|----------|
| 测试用例 | 与子代理的压力场景 |
| 生产代码 | 技能文档（SKILL.md）|
| 测试失败（RED）| 没有技能代理违反规则 |
| 测试通过（GREEN）| 有技能代理遵守 |
| 重构 | 关闭漏洞同时保持合规 |

#### 技能结构

**Frontmatter（YAML）：**
```yaml
---
name: skill-name-with-hyphens
description: Use when [具体触发条件和症状]
---
```

**内容：**
1. 概述
2. 何时使用
3. 核心模式
4. 快速参考
5. 实施
6. 常见错误
7. 实际影响

#### RED-GREEN-REFACTOR 循环

**RED：编写失败测试（基线）**
- 在没有技能的情况下与子代理运行压力场景
- 记录准确行为和合理化

**GREEN：编写最小技能**
- 写技能处理那些特定合理化
- 用技能运行相同场景
- 代理现在应该遵守

**REFACTOR：关闭漏洞**
- 代理找到新合理化？添加明确反驳
- 重新测试直到防弹

#### 技能创建清单

使用 TodoWrite 为每个清单项创建待办事项：

**RED 阶段：**
- [ ] 创建压力场景
- [ ] 无技能运行场景 - 记录基线行为
- [ ] 识别合理化模式

**GREEN 阶段：**
- [ ] 名称仅使用字母、数字、连字符
- [ ] YAML frontmatter
- [ ] 描述以"Use when..."开头
- [ ] 清晰概述
- [ ] 处理基线失败
- [ ] 有技能运行场景 - 验证遵守

**REFACTOR 阶段：**
- [ ] 识别新合理化
- [ ] 添加明确反驳
- [ ] 构建合理化表
- [ ] 创建红旗列表
- [ ] 重新测试直到防弹

**部署：**
- [ ] 提交技能到 git
- [ ] 考虑通过 PR 贡献回去

## 核心开发流程

### 完整功能开发流程

```
1. brainstorming（头脑风暴）
   ↓
2. writing-plans（编写计划）
   ↓
3. using-git-worktrees（创建隔离工作空间）
   ↓
4. executing-plans 或 subagent-driven-development（执行）
   ↓（每个任务）
   test-driven-development（TDD）
   ↓（遇到错误）
   systematic-debugging（系统调试）
   ↓（任务完成）
   verification-before-completion（验证）
   requesting-code-review（请求审查）
   receiving-code-review（处理反馈）
   ↓
5. finishing-a-development-branch（完成分支）
```

### 快速参考决策树

**开始新功能？** → brainstorming
**有多步骤任务？** → writing-plans
**执行计划？** → executing-plans 或 subagent-driven-development
**写代码？** → test-driven-development
**遇到错误？** → systematic-debugging
**声称完成？** → verification-before-completion
**需要审查？** → requesting-code-review
**收到反馈？** → receiving-code-review
**完成工作？** → finishing-a-development-branch

## 最佳实践

### 1. 始终遵循技能
如果技能适用，**必须**使用它。这不是可选的。

### 2. 先验证后声称
永不在运行验证命令前声称某事工作。

### 3. TDD 无例外
先写测试。看它失败。然后写代码。

### 4. 系统调试
找到根本原因后修复。症状修复是失败。

### 5. 增量工作
小步骤、频繁提交、持续验证。

### 6. 有效协作
请求审查、给出技术反馈、迭代改进。

## 常见陷阱

❌ 跳过技能因为"太简单"
❌ 在验证前声称完成
❌ 写代码后写测试
❌ 在理解根本原因前修复症状
❌ 在审查者错时盲目实施反馈
❌ 跳过代码审查因为"只是小改动"

## 总结

Superpowers 提供了一套经过验证的工作流程和最佳实践。通过遵循这些技能，你可以：

✅ 做出更好的设计决策（brainstorming）
✅ 创建清晰可执行的计划（writing-plans）
✅ 高质量执行（executing-plans、subagent-driven-development）

✅ 确保质量（verification-before-completion）
✅ 协作审查（requesting/receiving-code-review）
